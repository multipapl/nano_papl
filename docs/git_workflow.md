# Git Workflow Guide (Solo Developer)

Цей гайд допоможе тримати історію чистою, а релізи — організованими.

## 1. Щоденна робота (Гілка `dev`)
Працюйте тут постійно. Можете робити багато дрібних комітів.

```bash
# Створити гілку dev (якщо ще немає)
git checkout -b dev

# Додавання змін
git add .
git commit -m "feat: почав робити нову вкладку"
# ... через годину ...
git add .
git commit -m "fix: виправив баг з коліром"

# Бекап на GitHub (опціонально, для безпеки)
git push origin dev
```

## 2. Підготовка до релізу
Коли ви закінчили і хочете "випустити" версію:

1.  Оновіть версію в `main.py` (наприклад, `v1.3`).
2.  Зробіть білд (EXE).
3.  Запушіть `dev` на GitHub (про всяк випадок).
4.  Об'єднайте все в `main` одним комітом (**Squash**).

```bash
# Переходимо в main
git checkout main

# Об'єднуємо всі коміти з dev в один (squash)
git merge --squash dev

# Робимо чистий релізний коміт
git commit -m "Release v1.3: Нові фічі та виправлення"

# Додаємо тег для GitHub Releases
git tag v1.3

# Пушимо все
git push origin main --tags
```

## 3. GitHub Releases (Бест-практіс)
*   Не публікуйте реліз відразу. Створіть **Draft**.
*   Залийте туди свій EXE файл.
*   Якщо протягом дня з'явилися правки — просто перезалийте EXE в той самий Draft і оновіть коміт.
*   Натискайте **Publish** лише тоді, коли день закінчився і ви задоволені результатом.

---
**Порада:** Якщо ви випадково зробили коміт "Release v1.2", а потім зрозуміли, що треба ще одна правка — не бійтеся використовувати:
`git commit --amend` — це додасть ваші нові зміни до ПОПЕРЕДНЬОГО коміту, не створюючи нового "сміття".
